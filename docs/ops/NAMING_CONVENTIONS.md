# ACEE Naming Conventions (Legal • Platform • Product)

**Status:** Canon v1 (Immutable baseline)  
**Last updated:** 2025-12-13  
**Owner:** ACEE Engineering  
**Applies to:** All ACEE codebases, infrastructure, documentation, and AI agents (Cursor/RAG)

---

## 0) Purpose

This document defines the **single source of truth** for how we name the company, platform, products, repos, services, environment variables, documents, and identifiers.

**Goal:** Eliminate naming drift across repos, infrastructure (Render/Vercel), and AI-generated changes.  
**Non-goal:** Cosmetic renaming of existing runtime services/URLs unless explicitly planned.

---

## 1) Naming Layers (Authoritative)

We use a strict 3-layer model:

### 1.1 Legal Entity (Legal layer)
- **Legal name:** **ACEE Ventures LLC** (Delaware)
- Use only in:
  - Contracts, invoices, banking, tax
  - Terms of Service / Privacy Policy
  - Investor materials / compliance disclosures
  - Formal press statements
- Do **NOT** use in:
  - Repo names, code identifiers, env var prefixes, service names

### 1.2 Platform / System (Engineering top-level)
- **Platform/system name:** **ACEE**
- Use in:
  - Infrastructure namespaces (Render Env Groups, CI, automation)
  - Canonical environment variables for cross-service configuration
  - Ops docs and governance docs
  - AI rules and memory (RAG canon)

### 1.3 Product / Application (Product layer)
- **Product name:** **ProofOfInfluence**
- Use in:
  - UI, marketing, product documentation
  - Product-specific modules and features
  - User-facing routes/pages and product-specific references

**Canonical statement (must appear in top-level docs):**  
> *ProofOfInfluence is a core product module under the ACEE platform.*

---

## 2) Slugs & Case Rules

### 2.1 Canonical slugs
- Platform slug: `acee`
- Product slug: `poi` (short) or `proof-of-influence` (long, docs only)

### 2.2 Case rules
- **Repo names:** kebab-case (e.g., `acee-contracts`)
- **Service/env group names:** kebab-case (e.g., `acee-poi-prod-auth-secrets`)
- **Env vars:** UPPER_SNAKE_CASE (e.g., `ACEE_API_BASE_URL`)
- **Code identifiers:** camelCase / PascalCase per language conventions
- **DB:** snake_case (columns, indexes), plural table names unless clearly singular entities

---

## 3) Repository Naming

### 3.1 Required repos (v1)
- `ACEE` (meta / governance / standards / templates)
- `ProofOfInfluence` (platform + product module)
- `acee-games` (WS + engines, optional deployment)
- `acee-contracts` (public contracts)

### 3.2 Repo README naming rule
Each repo README must start with:
- "This repository is part of the **ACEE platform**."
- If product-specific: "This repository implements the **ProofOfInfluence** module."

---

## 4) Infrastructure Naming (Render / Vercel / DNS)

### 4.1 Render: Env Groups (must use ACEE top-level)
Env groups must be namespaced by platform + product slug + env + purpose:

**Pattern:**  
`acee-<module>-<env>-<purpose>`

Where:
- `<module>`: `poi` (ProofOfInfluence module), `games` (acee-games), etc.
- `<env>`: `dev` | `staging` | `prod`
- `<purpose>`: `db` | `auth-secrets` | `integrations` | `nonsecret-config` | `service-to-service`

**Examples (ProofOfInfluence):**
- `acee-poi-prod-db`
- `acee-poi-prod-auth-secrets`
- `acee-poi-prod-integrations`
- `acee-poi-prod-nonsecret-config`

**Forbidden:**
- Any group name starting with `proofofinfluence-` (legacy top-level)
- Any group containing `VITE_*` keys
- Any group containing dev bypass flags (see §6.4)

### 4.2 Render: Services
Render service names may remain as-is for stability (e.g., `proofofinfluence`) unless we plan an explicit rename window.

**Docs must refer to the service as:**  
"ACEE (ProofOfInfluence module) web service: `proofofinfluence`".

### 4.3 Vercel: Projects
Vercel project name can remain as-is. The authoritative naming is in docs + env vars.

### 4.4 DNS / Domains
- Use **ACEE** as the primary brand on domains.
- Recommended:
  - Web: `https://<brand-domain>` (ACEE)
  - API: `https://api.<brand-domain>`
  - WS: `wss://ws.<brand-domain>` (if needed)

## 5) Environment Variable Naming (Canonical)

### 5.1 Canonical rule
- Cross-service / top-level env vars use **`ACEE_` prefix**.
- Product-specific vars may use **`POI_`** *only* if the variable is exclusively used inside the ProofOfInfluence module and never shared.

### 5.2 Preferred canonical keys (examples)
- `ACEE_ENV` = `production|staging|development` (optional but recommended)
- `ACEE_WEB_BASE_URL` = public website base URL
- `ACEE_API_BASE_URL` = API base URL (Render)
- `ACEE_WS_BASE_URL` = WS base URL (if applicable)

### 5.3 Frontend env vars
Frontend build-time variables must be prefixed with:
- `VITE_` (for Vite projects)

Do **not** place `VITE_*` keys in Render.

### 5.4 Dev flags
Dev bypass flags must **never** be in env groups and must be **false/unset** in production.

Examples (non-exhaustive):
- `DEV_MODE_ADMIN`
- `DEV_MODE_AUTH_BYPASS`
- `AUTH_BYPASS`
- `ENABLE_DEV_FEATURES`
- `ALLOW_UNAUTHENTICATED_SETTLEMENT`

## 6) Deprecations & Backward Compatibility

### 6.1 Policy
- Old names may exist as **deprecated aliases** only.
- Canonical names are the only names used in:
  - docs
  - new code
  - new infra configuration

### 6.2 Implementation
- All alias mapping must live in **one file** (`envAliases.ts`) and warn-once when used.
- Deprecations must be recorded in `ENV_DEPRECATIONS.md` with:
  - canonical key
  - deprecated key(s)
  - removal deadline
  - migration notes

### 6.3 Deprecation example
- `BASE_URL` (deprecated) → `ACEE_API_BASE_URL` (canonical)
- `PROOFOFINFLUENCE_BASE_URL` (deprecated) → `ACEE_API_BASE_URL` (canonical)

## 8) Quick Migration Checklist (for Cursor / Devs)

When migrating naming:
1. Replace infra/documentation top-level references:
   - `proofofinfluence-*` → `acee-poi-*`
2. Keep runtime service names stable unless explicitly planned.
3. Ensure env groups contain no dev flags and no `VITE_*` keys.
4. Ensure canonical env vars use `ACEE_*` where appropriate.
5. Add/maintain alias mappings for any renamed env keys.
6. Run:
   - `npm run env:drift-check`
   - `npm run env:check --mode=ci web`
